# STEP 2 — Проверка регистрации и ролей

Дата проверки: 2026-05-23

## 1) Что проверено

### Регистрация / onboarding
- Проверен основной flow регистрации через `app/handlers/start.py`:
  - `/start` (`handle_start`)
  - шаг имени (`handle_registration_name`)
  - шаг даты рождения (`handle_registration_birthdate`)
  - шаг телефона/контакта (`handle_registration_phone`)
  - завершение регистрации + переход в главное меню (`_show_main_menu` / `render_main_by_role`)
- Проверено поведение `/start` во время FSM регистрации (`handle_start_during_registration`).
- Проверены состояния FSM: `REG_NAME`, `REG_BIRTHDATE`, `REG_PHONE`.
- Проверены валидации:
  - имя: минимум 2 символа
  - дата: `ДД.ММ.ГГГГ`, диапазон `1900-01-01 .. today`
  - телефон: нормализация + проверка валидности
- Проверено, что повторный `/start`:
  - для обычного зарегистрированного пользователя открывает меню без повторной регистрации
  - для разработчика может перезапустить регистрацию в debug-режиме
- Проверено, что незавершённый пользователь не блокируется: при ошибке синхронизации YClients FSM очищается, пользователь может перезапустить flow через `/start`.

### Роли, меню, доступ
- Проверена защита developer ID `378881880` через:
  - `app/core/permissions.py` (`DEVELOPER_TG_ID`, `resolve_role`)
  - `app/core/auth.py` (`normalize_role`, `is_protected_developer`, `can_manage`)
  - `app/repositories/users.py` (`ensure_protected_developer` вызывается в startup)
- Проверено рендеринг-ветвление главного меню по ролям через `app/keyboards/factory.py` + `app/keyboards/menu.py`.
- Проверены ограничения regular user:
  - отсутствие staff/admin/developer кнопок
  - невозможность handler-level доступа к защищённым разделам через декоратор `require_roles` и проверку роли в handler’ах.
- Проверены admin/manager ограничения по текущей проектной логике:
  - администратор не developer
  - developer-only элементы не доступны admin/manager
  - запрет управления защищённым developer

### Callback safety
- Проверены callback_data для меню/роль/onboarding:
  - длина <= 64 байт
  - базовая корректность формата и обработка malformed callback в `role_onboarding`.

## 3) Риски и замечания

1. В проекте есть альтернативный flow в `app/handlers/onboarding/registration.py`, который не подключён в `app/handlers/__init__.py`.
   - Это не runtime-баг текущего пути, но потенциальный риск расхождения логики при будущих изменениях.

2. Часть проверок выполнена на code-level и unit/integration test-level, а не в живом Telegram runtime.
   - Поэтому Telegram runtime-поведение должно быть дополнительно проверено вручную (см. MT ниже).

## 4) Что нужно проверить вручную (обязательно)

> Ниже только ручные сценарии. Они **не** отмечены как PASS автоматически.

### MT-001 — Регистрация нового пользователя
1. Открыть бота с новым Telegram-аккаунтом.
2. Выполнить `/start`.
3. Пройти шаги: имя → дата рождения → контакт/телефон.
4. Убедиться, что при успешном завершении показывается `✅ Регистрация завершена!` и открывается `🏠 Главное меню`.
5. Повторно отправить `/start`.
6. Убедиться, что повторная регистрация не запускается, открывается обычное меню.

### MT-002 — Невалидные данные регистрации
1. Запустить `/start` новым пользователем.
2. На шаге даты ввести невалидные варианты (`31-01-1999`, `31.02.1999`, будущую дату).
3. Убедиться, что бот корректно просит повторить ввод даты и не падает.
4. На шаге телефона отправить невалидный текст вместо номера.
5. Убедиться, что бот просит отправить контакт/номер повторно.

### MT-003 — Роли и доступ (developer/admin/manager/user)
1. Под developer ID `378881880` открыть `🏠 Главное меню`.
2. Убедиться, что отображается developer-ветка меню.
3. Под regular user убедиться в отсутствии restricted-разделов (`👥 Персонал`, `📊 Статистика`, `📣 Рассылка`, developer-only).
4. Под admin/manager проверить доступ только к разрешённым разделам по текущей продуктовой логике.

### MT-004 — Прямой вызов restricted callback
1. Под regular user попытаться вручную вызвать callback из staff/broadcast/developer секций (через тестовый клиент/скрипт).
2. Убедиться, что доступ блокируется на handler-level (не только скрытием кнопки в UI).
3. Проверить, что бот отвечает корректным сообщением об отсутствии прав без stack trace и без role degradation.
